   Antes deu aprovar o draft.yml preciso retirar uma dúvida: porque é necessário ter uma entidade Team │ Multitenancy? Pois esse
sistema não será utilizado por várias empresas não. Esse sistema será para uma empresa só controlar seus contratos e os clientes
dela. Ou seja, estou fazendo esse sistema exclusivamente para uma pessoa que me contratou para fazer. Talvez o multi-tenancy faça
sentido para controlar o Portal do cliente, caso não seja possível implementar o isolamento de outra maneira maissimples.

● Ótima pergunta. Você tem razão — Team/Multitenancy é desnecessário para o seu caso.

  Por que eu adicionei (e por que não faz sentido)

  O CLAUDE.md menciona team_id no exemplo de modelagem, mas isso é um template genérico. Como o sistema é para uma única empresa,
  adicionar team_id em todas as tabelas seria complexidade sem benefício — filtros globais, scopes, seed de team em tudo, etc.

  Isolamento do Portal do Cliente

  O isolamento do portal é simples sem multitenancy:

  - O Client já tem portal_user_id (FK para User)
  - No painel do Portal, basta um scope: Client::where('portal_user_id', auth()->id())
  - O usuário do portal só vê contratos, documentos e boletos do seu Client
  - Isso é apenas um filtro por usuário autenticado, não multitenancy

  Ou seja: um Panel separado no Filament (/portal) com um middleware que resolve o Client pelo User logado. Simples e direto.

  O que muda no draft.yaml

  - Remover a entidade Team inteira
  - Remover team_id de todas as entidades
  - Manter portal_user_id no Client para o portal
